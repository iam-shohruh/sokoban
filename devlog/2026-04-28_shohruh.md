Designed the solver architecture and project structure.

**Done:**
- Decided on `eval.py` as the orchestrator (not `solve.py` — avoids name clash with solver algorithms that each expose their own `solve()` function)
- Defined the data flow: `levels.json` → `state.py` → `eval.py` → solver algorithm → `results/`
- Solver algorithms live in `search/` (BFS, A\*, IDA\*, …) and `RL/`. Each exposes a `solve(state) -> list[Action] | None` function and calls `env.py` methods internally. `eval.py` picks the right one by name via a registry dict.
- `eval.py` is functional (one `run()` + private helpers), not a class — no polymorphism needed here
- Results written to `results/<dataset>/<solver>/levels.json` + `summary.json`
- Updated README with project structure tree and architecture diagram

**Decisions:**
- `eval.py` not `bench.py` — eval fits both search and RL contexts; bench implies only measuring speed
- `save_results` not `log_results` — log implies appending to a stream, save implies writing a file
- Env is used inside solver algorithms directly, not injected by the orchestrator — keeps `eval.py` decoupled from env internals
